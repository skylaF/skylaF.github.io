1. 域名系统DNS

DNS报文使用UDP数据报  
处理IP数据报时使用IP而不是域名，是因为IP地址时固定的32位，而域名不是固定的，所以机器处理起来比较难  
域名系统是分层次的：  
  - 根域名服务器：采用任播技术，一个请求报文过来，就会发到最近的根服务器，返回顶级域名服务器的地址  
  - 顶级域名服务器：  
  - 权限域名服务器：负责一个区中的所有域名和IP的映射  
  - 本地域名服务器：一个DNS客户发出请求后，就是发到这个服务器，相当于一个单位的域名服务器

  • 域名解析过程：

    • 递归查询：就是代替前一个查询客户去向其他服务器查询
    • 迭代查询：服务器收到请求报文，要么直接给出下一个服务器的地址，要么报错，交由客户自己去查询
      域名解析过程

  • 高速缓存：最近有查询过的域名以及从何处获得域名映射信息的记录，如果本地有查询报文的顶级域名IP地址，则不会向根服务器查询，而是之间向顶级服务器查询

2. 超文本传送协议HTTP

HTTP是面向事务的，有HTTP请求报文和HTTP响应报文；  
使用了TCP连接保证了可靠传输，但是HTTP本身是无连接、无状态的；  
经过一个RTT时间后，浏览器将HTTP请求报文作为TCP连接的第三次握手报文中的数据，发送给服务器；  
服务器收到之后就将所请求的文档作为响应报文返回，所以总共需要2个RTT时间，一个用于连接，一个用于请求和接收文档


HTTP响应过程

  • HTTP1.0：每次连接都要两倍的RTT时间，每个请求都要一个新的TCP连接，这种称为短连接

  • HTTP1.1：使用了持续连接，在一次响应后仍保持着连接，只要后续请求的文档在同一台服务器上，就能使用这一个连接

    • 非流水线方式：客户收到前一个响应之后才能发起下一个请求，每次请求都要用去一个RTT时间，只比短连接少了一个RTT的开销
    • 流水线：在响应之前就能接着发送请求，因此访问所有对象只需要一个RTT时间

  • 2.1. HTTP报文首部 两种格式的报文其实就是首部开始行的不同

    • 请求报文首部
      • 请求行（开始行）：
        • 方法：GET、POST、DELETE等
        • URL：
        • HTTP的版本：
      • 首部行：host地址
      • 主体
    • 响应报文首部
      • 状态行（开始行）：状态码、HTTP版本、解释状态码
        状态码

    常用状态码：

    • 200：成功
    • 202：接受
    • 301：永久性重定向
    • 400：错误请求
    • 401：认证失败
    • 403：禁止
    • 404：找不到

  • 2.2. Cookie

    • 具体信息存放在服务器上
    • cookie存放在客户浏览器上
    • 由服务器为客户产生一个识别码，设定Set-cookie字段，值设为这个识别码，发给客户，浏览器将其添加进本地cookie文件中
    • 此后该用户浏览这个网站时，都会带上这个Cookie信息在首部行中，一遍服务器判断出这个用户是谁。

  • 2.3. 其他实现HTTP有状态的方式

    • 2.3.1. session：每次会话都有一个特定的sessionId，这个Id可以用来为之后的其他会话记录信息，服务器可以将这个id对应的信息存在服务器的数据库上，方便以后识别用户, session是由服务器创建的，存储在服务器端，然后将sessionID交给浏览器，浏览器将其存储在cookie中，每次发送报文就放在cookie字段的值域里，所以session也是依靠cookie使用的，如果浏览器禁用Cookie，session只能使用url的方式发送给服务器
    • 2.3.2. JWT:json web token，也是用来实现用户跟踪的，由服务器创建，并发送给浏览器，客户端存储这个token，然后每次访问就带上这个token。
      • JWT本质是一个json对象，由三个部分组成：头部、载荷、签名,三部分由“ . ”隔开。
      • 头部：描述这个JWT使用的东西，比如签名是用什么方式，这个令牌的类型
      • 载荷：就是你要传递的数据，有7个官方字段可供使用，也可以定义私有字段，用来传递用户信息什么的
      • 签名：对前两部分加密，可以使用SHA256算法
      • 将三部分拼成一个字符串，就可以发给用户了，用户请求的时候直接放在首部字段里，这样可以跨域使用

3. 文件传送协议FTP

使用TCP的运输服务，服务器进程由两大部分组成：一个**主进程**，负责接收新的请求；一个**从属进程**，负责处理单个请求
  • 3.1. 主进程工作步骤如下：
    • 主进程监听21端口，等待客户吨发起连接请求；
    • 启用从属进程处理客户进程发来的请求，处理完毕之后即终止，但从属进程运行期间可能创建其他一些子进程。
  • 3.2. 从属进程包括控制进程和数据传送进程，也就是两个TCP连接
  • 3.3. 具体过程如下:
    • 客户端要连接时，寻找服务器的21端口，同时告知服务器自己的另一个端口号，用于数据连接，接着服务器用自己的20端口与这个端口进行连接。（这里有个问题就是，如果都是用21和20端口，那么多用户时是怎么处理的？）
  • 两种模式（根据是否是服务器主动连接的）：
    • 主动模式：服务器使用21和20端口，客户端端口随机
    • 被动模式：客户端连接到服务器的21端口，服务器数据连接的端口随机。

5. HTTPs

HTTPs = HTTP + 加密 + 认证 + 完整性保护


http和https

  • 5.1. 使用https的原因
    • http明文通信，内容可能被窃听
    • http不验证通信方的身份
    • http无法证明报文的完整性，可能遭到篡改
  • 5.2. 如何保证信息的加密
    • 5.2.1. 对称加密 加密和解密使用同一个密钥
    • 5.2.2. 非对称加密 加密和解密使用不同的密钥，称为公钥和私钥，公钥是给对方加密的，私钥是用来解密对方信息的
  • 5.3. 如何保证完整性
    使用信息摘要，比如MD5，信息摘要，就是将报文内容通过甲酸得到一串哈希值，这个哈希值就是摘要，接收方只要在接收到报文之后再次计算报文内容，将得到的哈希值与发送方的哈希值进行就能知道是否完整了。
  • 5.4. HTTPs采用混合加密的方式
    因为使用非对称方式的加密的处理速度比较慢，所以采用先公开密钥的方式交换对称密钥信息，然后后续使用对称密钥进行通信

6. 动态主机配置协议DHCP

  • DHCP报文只是UDP数据报的数据，还要加上UDP首部、IP数据报首部、以太网的MAC帧首部和尾部之后才能传送
  • 允许一台计算机自动获取IP地址，DHCP也是采用C/S方式；
  • 需要获取IP地址的主机在启动时就向DHCP服务器广播发送发现报文，服务器现在数据库中查找这个主机的IP地址，如果没有就从IP池中随机分配一个地址给这台主机，返回提供报文；
  • 实际上，并不是主机自己向DHCP服务器发送发现报文，而是通过DHCP中继代理（一台路由器，保存在DHCP服务器的IP地址）以单播方式向服务器转发此报文。
  • DHCP分配的IP地址是临时的
