1. 进程状态


进程状态转换图

    1. 创建：申请PCB，分配资源，放进就绪队列，只要资源还没分配好，就一直再创建状态
    1. 终止：用来将不能再执行或者执行完毕的进程资源回收，PCB清零，空间返还系统，或者等待其他进程收集该进程的状态码和一些计时统计数据。
    1. 就绪：就绪队列中的进程在等待被调度，一旦调度，就进入执行状态。
    1. 执行：正在执行中的程序。
    1. 阻塞：正在执行的某个进程发生I/O请求、申请缓冲区失败等事件时而无法继续执行的状态。
    1. 引入挂起操作：多了静止就绪、静止阻塞两个状态，如下：
      挂起操作的状态图(I/O请求应该是执行指向活动阻塞)

2. 进程的数据结构

  • 2.1. 进程控制块PCB
    • 进程标识符（内部标识符，os使用；外部标识符，用户/进程使用）
    • CPU上下文（现场信息：通用寄存器、指令计数器、程序状态字PSW）
    • 进程调度信息（进程状态、进程优先级）
    • 进程控制信息（进程同步和通信机制、程序和数据地址、资源清单、链接指针）
  • 2.2. PCB组织方式
    • 线性方式：存储在整一张表中，每次查询都要搜索整张表（按创建先后排列）
    • 链接方式：相同状态的PCB链接成一张链表
    • 索引方式：每一种状态都建立一张索引表，这张表按顺序存储处于这种状态的PCB的指针，类似于二级表

3. 进程同步

保证并发执行的程序的可再现性，使并发执行的进程之间按照一定的规则（时序）共享系统资源。  
一般来说，只要一个资源能被多个访问，就需要加互斥量

3.1. 同步规则

  • 空闲让进
  • 忙则等待
  • 有限等待：避免死等
  • 让权等待：当进程不能进入临界区，立即释放CPU，以免忙等

3.2. 同步工具（1）——信号量机制

信号量是一种进程同步工具
    1. 整型信号量
      有一个用来表述资源数目的整型S，仅能通过两个原子操作wait和signal进行操作，即PV操作缺点在于S<=0时需要不停的测试，不满足让权等待的规则 
      wait(S){
  while(S<=0);
  S--;
}
signal(S){
  S++;
}

      P操作：若信号量大于0，则允许进程继续执行，并将信号量减一；若信号量小于等于0，则进程被加入到同步队列中等待，直到信号量大于0时被唤醒。通常用于实现互斥操作或限制进入临界区的进程的数量。
      V操作：对信号量加一，若加一后信号量大于0，则唤醒在同步队列中等待的进程。

    1. 记录型信号量
      增加了一个等待进程链表，并且采取了让权等待的策略 
      wait(semaphore *S){
  S->value--;
  if(S->value<0) block(S->list);
}
signal(semaphore *S){
  S->value++;
  if(S->value>0) wakeup(S->list);
}
    1. AND信号量
      以上两种信号量都是针对一个临界资源的情况，AND信号量解决多个临界资源的情况，多个临界资源可能导致死锁，而AND信号量可以避免死锁，方法就是：将进程运行中所需的所有资源一次性的分配给他，反之如果有一个资源不能分配给它，其他所需资源也不能分配给它。 
      Swait(S1,S2,...,Sn){
  while(true){
    if(Si>=1&&...&&Sn>=1{
      for(i=1;i<=n;i++)Si--;
      break;
    }
    else{
      //进入等待队列，直到满足分配条件
    }
  }
}
Ssignal(S1,S2,...,Sn){
  while(true){
    for(i=1;i<=n;i++){
      Si++;
      //从等待队列中移除所有进程
    }
  }
}
    1. 信号量集
      分配时不再是一个单位资源的分配，而是n个，对资源的检测也不是1，而是下限值了 当下限值为1，需求值为1时，即为一般的记录型信号量

3.4. 同步工具（2）-管程monitors

信号量机制的缺点是，对每一个信号量的P,V操作都需要按照严格的顺序配对出现，当信号量数量较多时容易出错，难以维护。  
所谓管程，指的是管理共享变量以及对共享变量的操作过程，让他们支持并发  
管程是封装共享资源以及对资源进行操作的一组过程的模块；由四部分组成：  
  - 名称，  
  - 共享数据结构说明；  
  - 对该数据结构进行操作的一组过程；  
  - 对共享数据初始化的语句
    1. 管程的特点
    • 进程访问临界资源时只能通过管程间接访问
    • 每次只允许一个进程进入管程，执行管程内的过程
    • 管程内的数据结构只能被管程内的过程所访问
    1. 条件变量
      为了防止一个进程在管程中被阻塞或挂起而一直占用管程导致其他进程无法进入管程
    • 条件变量 = 信号量 + 等待队列（链表）
    • 条件变量也提供两个操作：
      • cwait(condition)：使调用进程阻塞，加入等待队列中等待唤醒
      • csignal(condition)：唤醒在该条件的等待队列中的进程
    • Java的管程例子 
      public class MonitorExample {
    /**
     * 共享资源
     */
    private int resource;

    /**
     * 条件变量
     */
    private final Object condition = new Object();

    /**
     * 递增操作，返回递增后的结果
     */
    public int incr() throws InterruptedException {
        synchronized (condition) {
            while (resource >= 10) {
                condition.wait();//多于上限值就等待消费  
            }
            resource++;
            condition.notifyAll();
        }
        return resource;
    }

    /**
     * 递减操作，返回递减后的结果
     */
    public int desc() throws InterruptedException {
        synchronized (condition) {
            while (resource <= -10) {
                condition.wait();//低于下限值就等待生产  
            }
            resource--;
            condition.notifyAll();
        }
        return resource;
    }
}
